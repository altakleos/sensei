# Release Process

## Intent

Sensei is distributed as a Python package on PyPI. The release process is the point at which code on `main` becomes an installable version users depend on. This spec captures the properties every public release must satisfy regardless of how the release workflow is implemented. It is the product-level contract; the mechanism (what CI runs, which build backend, which authentication method) is a design concern documented separately.

## Invariants

- **Reproducibility from tag.** Every public release is built from a single tagged commit on `main`. Anyone with the tag can rebuild an equivalent artifact from source.
- **Semver.** Release tags match the pattern `v<major>.<minor>.<patch>` with optional pre-release suffix (`-alpha`, `-beta`, `-rc.N`, `a1`, `b2`). No ad-hoc version strings.
- **Version concordance.** The version declared in `src/sensei/__init__.py.__version__` exactly matches the tag on the commit being released. A mismatch is a release-blocking error.
- **Integrity of the engine bundle.** The published wheel contains the engine bundle (`sensei/engine/` with `engine.md`, `defaults.yaml`, `protocols/`, `scripts/`, `prompts/`, `schemas/`, `profiles/`) complete. Missing any of these files is a release-blocking error.
- **No instance content.** The published wheel MUST NOT contain any directories or files that belong to user instances: `instance/`, `wiki/`, `raw/`, `notebook/`, `inbox/`, `memory/`, or any tool-specific shim generated by `sensei init` (`CLAUDE.md`, `.cursor/`, `.kiro/`, etc.). Shipping instance content is a release-blocking error.
- **Pre-publish verification.** A release is publish-eligible only if the full test suite passes, the wheel builds, and every invariant above is mechanically checked. A failing check blocks publishing; no human override at the publish step.
- **Ambient authentication.** Publishing uses short-lived, ambient credentials (e.g., OIDC trusted publishing). Long-lived PyPI API tokens stored as secrets are not permitted. Rotating authentication must not require editing CI configuration.
- **Human approval at publish.** Publishing to PyPI requires an explicit approval step by a designated reviewer between the passing verify gate and the upload. Automatic, un-gated publishes are not permitted.
- **Atomic publish.** A publish either succeeds (wheel and sdist available on PyPI) or leaves PyPI in its prior state. Partial uploads that produce a half-released version are a release-blocking error.
- **Idempotent re-runs.** Re-running the release workflow for a tag that has already been published produces a no-op (or a duplicate-upload rejection by PyPI), never a silently-different second artifact at the same version.
- **Yank, don't delete.** Defective releases are yanked on PyPI; they are not deleted. Yanking preserves installability for anyone who pinned that version while hiding the release from unpinned `pip install`.
- **Instance upgrade preservation.** A release that changes the engine bundle does not corrupt existing instances. Users explicitly trigger `sensei upgrade` to adopt a new engine; no network request or auto-upgrade is ever initiated by the installed CLI.

## Rationale

**Reproducibility, tag concordance, and bundle integrity** exist to make every release auditable. A future contributor can rebuild an old release from the tag and get the same bytes; a support request about "version X does Y" can be verified by installing X and reading its contents. Without these, the product is a moving target.

**Instance-content exclusion** protects against a specific failure mode: accidentally shipping a maintainer's working instance inside the wheel, which would overwrite user content on install. The check must be mechanical because human review has a poor track record on "is this file supposed to be in the wheel."

**Ambient authentication with human approval at publish** is the 2024–2026 PyPI supply-chain posture. Long-lived tokens in CI are the largest blast radius for a compromised workflow; OIDC rotates on every run. Human approval at the last step means a compromised CI cannot silently publish.

**Atomic + idempotent publishing** means release incidents are bounded. If a publish fails partway through, re-running picks up cleanly; a tag is either released or not, never ambiguously so.

**Yank-don't-delete** matches PyPI semantics and protects users who pinned. Deletion irreversibly breaks any lockfile with the deleted version pinned.

## Out of Scope

- The choice of CI provider (GitHub Actions vs. alternatives). The design doc picks one.
- The choice of build backend (hatchling — fixed by ADR-0004).
- The specific GitHub Environment name, reviewer roster, or approval UI.
- Artifact signing (Sigstore / PEP 740). Aspirational but not a v1 invariant.
- Automated changelog generation.
- Automated version-bumping (version bumps remain manual commits).
- Post-publish announcements, newsletter integration, or release notes automation.
