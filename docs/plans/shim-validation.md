# Plan: Shim Validation — Static Snapshot + Manual Runbook

`docs/foundations/vision.md:9` and `README.md:83-93` honestly admit that six of the eight bundled tool shims are unverified: Cursor, GitHub Copilot, Windsurf, Cline, Roo, AI Assistant. The shim files are generated by `src/sensei/cli.py:33-48` `_SHIMS`, but no test asserts the generated content matches each tool's documented instruction-file format, and no runbook exists for end-to-end behavioral validation.

What this plan delivers (and what it does NOT):

- ✅ **Static format validation in CI.** A snapshot test that pins the exact bytes generated for each shim path and the per-tool format expectations (Cursor needs `.mdc` with frontmatter; Windsurf needs `trigger:`; Copilot needs plain prose at a fixed path; etc.). Catches `_SHIMS` typos and accidental drift.
- ✅ **Manual validation runbook.** `docs/operations/shim-validation.md` — a per-tool checklist a maintainer can run on a machine that has the tool installed. One section per tool, each with: install the tool, scaffold an instance, open the folder, observe whether the agent picks up the boot chain, recordable pass/fail.
- ✅ **README truth-in-advertising.** Tier the table: ✅ "Verified" (Claude Code, Kiro), ⚙ "Format-validated; behavior unverified" (the six others). Today the table just says "Shim provided" which understates the gap on the unverified side.
- ❌ **NOT in scope: actual behavioral validation of the six tools.** Each is a GUI editor or IDE plugin without a headless CLI driveable from this environment. Behavioral validation needs a maintainer with the tool installed; this plan provides the runbook for that, not the run.

## Tasks

- [x] T1: `tests/test_init.py` — add `test_shim_<tool>_format_*` parametrised cases that scaffold a fresh instance and assert each generated shim file:
  - Exists at the expected path.
  - Contains the documented frontmatter or required directive for its tool (e.g. Cursor's `.mdc` parses as YAML frontmatter with both `description` and `alwaysApply` keys; Windsurf has a `trigger:` field with a documented value; Cline/Roo/Copilot are plain markdown referencing `AGENTS.md`).
  - References `AGENTS.md` (every shim's job is to point at the boot doc).
  - Catches the case where someone edits `_SHIMS` and accidentally drops the AGENTS.md reference, or breaks a frontmatter key the target tool requires.
- [x] T2: `tests/test_init.py` — add a snapshot test pinning the exact bytes of `_SHIMS` itself. Format: `_EXPECTED_SHIMS: dict[str, str] = { ... }` constant in the test file matching the dict in `cli.py`. Any change to `_SHIMS` then requires a deliberate snapshot update — no silent drift.
- [x] T3: Create `docs/operations/shim-validation.md` — manual runbook with one section per tool (Claude Code, Kiro, Cursor, GitHub Copilot, Windsurf, Cline, Roo, AI Assistant). Each section:
  - Tool name, install command/link.
  - "Scaffold a Sensei instance" step (`pip install sensei-tutor && sensei init <dir>`).
  - "Open the folder with the tool" step.
  - One probe prompt to send (e.g. "What do you do when the learner asks `Teach me about X`?").
  - Pass criteria: the agent's response references the boot chain or invokes a protocol-style behavior, NOT a generic chatbot reply.
  - Where to record the result (commit message convention or a dated note in the runbook).
  Mark Claude Code and Kiro as already validated; the other six get unchecked outcomes for the first maintainer to fill in.
- [x] T4: Update `README.md:83-93` shim table — add a "Behavior verified" column. Current "Verified" → ✅ in the new column for Claude Code and Kiro. The other six get "Shim format validated; behavior unverified — see [`docs/operations/shim-validation.md`](../operations/shim-validation.md)". Don't promise more than the runbook can deliver.
- [x] T5: Update `docs/foundations/vision.md:9` — the same truth-in-advertising adjustment, kept consistent with the README. Keep the language "shims provided and await further validation" if anything; the new runbook gives that validation a concrete path forward.
- [x] T6: `CHANGELOG.md` — append under `## [Unreleased]` → `### Added`:
  > Shim validation: every generated tool shim (Claude Code, Kiro, Cursor, Copilot, Windsurf, Cline, Roo, AI Assistant) is now format-asserted at CI time, and `docs/operations/shim-validation.md` provides a per-tool manual end-to-end runbook for behavioral verification. README and vision now distinguish "behavior verified" from "format validated, behavior pending".

## Out of Scope

- Behavioral end-to-end validation of the six unverified tools. Without headless drivers for those tools, this is operator work; the runbook makes it tractable but doesn't run it.
- Adding new tools to `_SHIMS`. Each new tool is its own decision (where does its instruction file live, what format does it need); separate concern.
- Auto-detecting which tool the user has installed and scaffolding a smaller shim set. Distribution is "all shims, every tool" by design — most cost is one file each.
- Pinning the exact byte length of `AGENTS.md` referenced by the shims. Shims point AT it; the boot doc has its own validation surface.
