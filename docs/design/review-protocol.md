# Review Protocol — Orchestration

## Context

The [`review-protocol`](../specs/review-protocol.md) spec defines what review guarantees: retrieval-only, stale-first, one topic at a time, no reteach. This design doc specifies how a single prose-as-code protocol realizes those guarantees by orchestrating four helpers (`check_profile`, `decay`, `classify_confidence`, plus an optional `mastery_check` as a gate for future extensions) over the learner's profile.

## Specs

- [review-protocol](../specs/review-protocol.md) — the guarantees this orchestration realizes
- [learner-profile](../specs/learner-profile.md) — the state it reads and writes

## Architecture

### File location

`src/sensei/engine/protocols/review.md`. Copied into `.sensei/protocols/review.md` in distributed instances via `sensei init`.

### Structure: single protocol, no sub-protocols

One file owns the full read → rank → ask → classify → write → re-validate loop. No `record_response.md` sub-protocol at v1. If a later protocol (e.g., assess, challenge) needs to share the write path, that refactor earns its own ADR.

### The orchestration (nine steps)

The protocol prose maps to these numbered steps. Each step is either a helper invocation (shell subprocess per ADR-0006) or an LLM-side operation. The LLM is responsible for literal step-by-step execution; it may not reorder, skip, or batch.

1. **Load and validate the profile.** Read `instance/profile.yaml`. Invoke `python .sensei/scripts/check_profile.py --profile instance/profile.yaml`. If exit ≠ 0, surface the error to the learner and end the session. Never proceed on an invalid profile.

2. **Enumerate stale topics.** For each topic in `expertise_map`, invoke `python .sensei/scripts/decay.py --last-seen <topic.last_seen> --half-life-days <config.memory.half_life_days> --now <current-utc> --stale-threshold <config.memory.stale_threshold>`. Collect the topics where `stale == true` along with their freshness scores.

3. **Exit if nothing is due.** If no stale topics remain after step 2, tell the learner "Nothing due for review right now." and end the session cleanly. No padding, no encouragement.

4. **Rank candidates by freshness ascending.** Sort the stale topics by `freshness` ascending (most stale first). Ties broken by earliest `last_seen`. This ranking is recomputed at the top of the loop only once per session — picking up a new topic after each successful write does not re-rank.

5. **Pose a single retrieval question for the top candidate.** Compose one question appropriate to the topic. No priming, no recap of the topic ("Last time we looked at recursion..."), no scaffolded hint. The question is generated by the LLM in prose; no helper proposes it at v1. Examples of acceptable shapes: "What does `map` do in Python?", "Give me the recurrence relation for merge sort's running time.", "Write the base case for factorial." The learner's response is the next message.

6. **Classify the response.** The LLM determines two binary labels from the learner's response:
   - `correctness` — `correct` or `incorrect` (did the learner produce the expected answer?)
   - `confidence` — `confident` or `uncertain` (judged from response style: direct and assertive → confident; hedged, partial, "I think…" → uncertain)

   Invoke `python .sensei/scripts/classify_confidence.py --confidence <label> --correctness <label>` to obtain the quadrant and short interpretation.

7. **Update the profile.** V1 update rule — deliberately minimal:
   - `last_seen` ← current UTC ISO-8601 timestamp
   - `attempts` ← `attempts + 1`
   - `correct` ← `correct + 1` if `correctness == "correct"`, else unchanged
   - `mastery` and `confidence` are **not touched** by review. Mastery promotions and confidence calibration are a future protocol's concern; a single retrieval event is too thin a signal to adjust either. This is a deliberate scope cut and will be revisited when a calibration protocol lands.

   Write the updated `profile.yaml`. Then invoke `check_profile.py` again. If the write produces an invalid profile, revert the change in memory, write the previous state back, and surface the error. This is the "every write validated" invariant from the spec.

8. **Continue or exit.** Check whether more stale topics remain in the ranking from step 4. If the learner expressed "stop" / "that's enough" / "later" in the previous turn, exit. Otherwise, loop to step 5 with the next candidate.

9. **Sign-off.** When the session ends (either naturally by exhausting candidates or on learner exit), acknowledge in one short line and stop. No summary, no celebration, no "great work today." The silence profile says less is more.

### Silence profile

Review sits on the assessor side of the silence model (§3.10). The mentor speaks to pose the question, acknowledge the answer, announce a transition, and stop. Neutral acknowledgments are permitted ("Got it.", "Okay.", "Next."). Praise, elaboration, and apology are not. The protocol prose must include explicit don'ts so the LLM does not drift.

### Config references

Review reads two tunables from the merged config (engine `defaults.yaml` + instance `config.yaml`):

| Key | Default | Role |
|---|---|---|
| `config.memory.half_life_days` | `7.0` | Per-topic half-life in days passed to `decay.py`. Controls how quickly freshness decays. |
| `config.memory.stale_threshold` | `0.5` | Freshness below this is considered stale. Same value is passed to `decay.py`. |

Both keys are added to `defaults.yaml` by T3 (implementation). No per-topic override at v1; the half-life is global. A future design may introduce per-topic calibration (the speed calibration from Math Academy §8.6 is the long-term target).

### Error handling

| Failure mode | Behaviour |
|---|---|
| Step 1: `check_profile` fails | Surface error; end session; no writes. |
| Step 2: `decay.py` fails for any topic | Treat that topic as not-stale (defensive skip); record in session log. If every invocation fails, end the session with a diagnostic. |
| Step 6: `classify_confidence.py` fails | End session; no writes. |
| Step 7: write invalidates the profile | Revert the write, surface the error, end session. The previous valid profile is preserved on disk. |
| Learner says "stop" mid-question | Do not write the in-flight response; exit cleanly. The interrupted topic is not penalized. |
| Subprocess missing Python / missing script | End session with a setup-error message pointing to `sensei verify`. |

### Interfaces

| Component | Reads | Writes |
|---|---|---|
| `protocols/review.md` (LLM) | `instance/profile.yaml`, `instance/config.yaml`, `.sensei/defaults.yaml` | `instance/profile.yaml` |
| `scripts/check_profile.py` | `profile.schema.json`, `profile.yaml` | stdout JSON |
| `scripts/decay.py` | (stateless; CLI args only) | stdout JSON |
| `scripts/classify_confidence.py` | (stateless; CLI args only) | stdout JSON |

### Engine dispatch

`engine.md` dispatch table is updated so the `review` signal (user intents like "review", "what should I review?", "let's review", "quiz me on what I've learned") routes to `protocols/review.md`. Status flips from `TBD` to `accepted`. The previous "Review my solution" code-review intent is re-assigned to a future `protocols/review-code.md` file — no code-review protocol exists at v1.

## Decisions

- [ADR-0006: Hybrid Runtime](../decisions/0006-hybrid-runtime-architecture.md) — protocols invoke helpers via shell subprocess; this design applies that pattern for the first time
- Review only touches `last_seen`, `attempts`, `correct`. Mastery and confidence updates are deferred to a future calibration protocol — documented in the step-7 comment above. A separate plan-scoped ADR may record this decision if the split proves durable.

## Notes

**Why nine steps and not six?** Earlier drafts merged steps (e.g., "ask and classify" as one step). Splitting keeps the protocol easy to verify in tests (each step has observable effects) and keeps the prose easier to write without tangled control flow.

**Why no re-ranking inside the loop?** Because the decay-based ordering is stable over a session: nothing the protocol does during a review session would change the relative freshness of other topics within the same minute. Re-ranking each iteration is wasted work.

**Why not derive `confidence` from ratios?** A future calibration protocol may compute confidence from `correct/attempts`. Review itself does not because the spec says review is retrieval-only, and inferring confidence from history would be a form of implicit calibration. Keep the concerns separated.
