# Release Workflow

## Context

The [`release-process`](../specs/release-process.md) spec defines the invariants every public release must satisfy — reproducibility, semver, version concordance, engine-bundle integrity, no instance content, ambient auth, human approval, atomicity, idempotency. This design specifies the concrete GitHub-Actions-based mechanism that realizes those invariants and the maintainer-side validator (`check_package_contents.py`) that makes the checks mechanical rather than aspirational.

## Specs

- [release-process](../specs/release-process.md) — the invariants this workflow enforces

## Architecture

### Workflow layout

Two workflow files under `.github/workflows/`:

1. **`verify.yml`** (already exists) — runs on every push and PR to `main`. Matrix across Python 3.10–3.13. Runs `pytest`. Not release-specific; catches regressions before they reach a tag.
2. **`release.yml`** (new) — runs on version tag push. Three jobs in strict order:
   - `verify` — matrix pytest across 3.10–3.13 on the tagged commit. Re-runs what `verify.yml` already ran on main; defence-in-depth because release is the last gate.
   - `build-and-check` — builds wheel and sdist; runs `check_package_contents.py` against the wheel asserting bundle integrity, no instance content, and version concordance with the tag.
   - `publish` — gated on GitHub Environment `pypi` approval; uploads via `pypa/gh-action-pypi-publish` using OIDC trusted publishing. Requires `id-token: write` permission.

### Tag filter

The `release.yml` `on.push.tags` filter matches semver and PEP 440 prerelease patterns:

```yaml
on:
  push:
    tags:
      - 'v[0-9]+.[0-9]+.[0-9]+'
      - 'v[0-9]+.[0-9]+.[0-9]+-*'
      - 'v[0-9]+.[0-9]+.[0-9]+[a-z][0-9]+'
```

Matches `v1.2.3`, `v0.1.0-alpha`, `v0.1.0-rc.1`, `v0.1.0a1`, `v0.1.0b2`. Unmatched tags do not trigger the workflow (e.g., `release-2026-04` is ignored).

### `check_package_contents.py`

Maintainer-side validator at `ci/check_package_contents.py` (repo root, **not** in the engine bundle — this is release-time tooling, not runtime). Takes a built wheel and an expected tag, opens the wheel as a zipfile, and asserts:

- **Required files present** (engine bundle complete):
  - `sensei/__init__.py`
  - `sensei/engine/engine.md`
  - `sensei/engine/defaults.yaml`
  - `sensei/engine/protocols/README.md`
  - `sensei/engine/protocols/review.md`
  - `sensei/engine/scripts/config.py`
  - `sensei/engine/scripts/check_profile.py`
  - `sensei/engine/scripts/mastery_check.py`
  - `sensei/engine/scripts/decay.py`
  - `sensei/engine/scripts/classify_confidence.py`
  - `sensei/engine/schemas/profile.schema.json`

- **Required directories present** (each must contain at least one entry):
  - `sensei/engine/prompts/`
  - `sensei/engine/schemas/`
  - `sensei/engine/profiles/`

- **Forbidden path prefixes absent** (no instance content leaked):
  - `instance/`, `wiki/`, `raw/`, `notebook/`, `inbox/`, `memory/`
  - `.kiro/`, `.cursor/`, `.windsurf/`, `.clinerules/`, `.roo/`, `.aiassistant/`, `.github/copilot-instructions.md`
  - `AGENTS.md`, `CLAUDE.md` at the wheel root (these are per-instance boot shims, generated by `sensei init`, never shipped)

- **Version concordance**: parse `sensei/__init__.py`, extract `__version__`, and assert it equals `tag` with the leading `v` and any leading PEP 440 dash/letter components stripped according to the matching tag form. Concretely: `v1.2.3` → `__version__ == "1.2.3"`; `v0.1.0-alpha` → `__version__ == "0.1.0a0"` OR `__version__ == "0.1.0-alpha"` depending on PEP 440 normalization; validator accepts PEP 440 equivalence. Mismatch is a hard failure.

Exit codes:
- `0` — all checks passed
- `1` — one or more required files/directories missing
- `2` — one or more forbidden entries present
- `3` — version concordance failure

Always prints a structured report to stdout (grouped by category: OK/MISSING/FORBIDDEN/VERSION). Tests under `tests/ci/test_check_package_contents.py` cover each failure mode against synthetic wheel fixtures.

### Environment protection

A GitHub Environment named `pypi` gates the `publish` job. The environment is configured (manually, outside this repo) with:
- **Required reviewers**: at least one designated maintainer.
- **Deployment branch and tag restriction**: **tag pattern `v*`**. The `release.yml` workflow is triggered by tag push, so the triggering ref is the tag itself, not `main`. A branch restriction on `main` would reject every tag-triggered run. The tag pattern must match (or be a superset of) the workflow's `on.push.tags` filter.
- **Wait timer**: 0 (approval is the gate, not delay).

### Authentication

Publishing uses OIDC trusted publishing. PyPI trusted-publisher registration (done manually on PyPI before the first release) associates the `altakleos/sensei` repository + `release.yml` workflow + `pypi` environment with the `sensei` project. No PyPI API tokens stored in GitHub Secrets.

### Atomicity

PyPI rejects duplicate uploads at the same version. If `publish` fails mid-upload and leaves PyPI in its prior state, re-running the workflow is safe — the upload either succeeds or is rejected as a duplicate. The `pypa/gh-action-pypi-publish` action handles this correctly out of the box.

### Interaction with existing `verify.yml`

`verify.yml` is unchanged by this design. It continues to run on pushes and PRs to `main`. `release.yml` does not invoke `verify.yml` as a reusable workflow; it has its own `verify` job with similar logic. The duplication is intentional — keeping `release.yml` self-contained means release-time behaviour cannot be accidentally broken by edits to `verify.yml`.

### Playbook integration

After this design lands and `release.yml` is in place, `docs/operations/release-playbook.md` has its aspirational preamble removed. The playbook's "Normal Release" section already describes the target flow; only the status line needs updating.

## Interfaces

| Component | Role |
|---|---|
| `.github/workflows/release.yml` | The three-job release workflow (new) |
| `.github/workflows/verify.yml` | Per-push verification (existing; unchanged) |
| `ci/check_package_contents.py` | Maintainer-side wheel validator (new) |
| `tests/ci/test_check_package_contents.py` | Unit tests for the validator (new) |
| GitHub Environment `pypi` | Human-approval gate (manual setup, outside repo) |
| PyPI trusted publisher for `sensei` | OIDC association (manual setup, outside repo) |

## Prerequisites (maintainer, outside this plan)

These must be done by a maintainer before the first real release can cut:

1. Register `sensei-tutor` on PyPI as a project (first publish can use `trusted-publishers/pending/` flow without an existing project). Distribution name per [ADR-0010](../decisions/0010-pypi-distribution-name.md).
2. Configure the PyPI trusted publisher: project `sensei-tutor`, repository `altakleos/sensei`, workflow `release.yml`, environment `pypi`.
3. Create the GitHub Environment `pypi` with a reviewer and the `main`-branch restriction.
4. Capture the environment ID in `docs/operations/release-playbook.md` (currently `<TBD>`).

## Notes

**Why duplicate the `verify` steps instead of reusing `verify.yml`?** Reusable workflows require adding a `workflow_call` trigger to `verify.yml` and coupling the two files. For a ~30-line workflow, the duplication is cheaper to maintain than the coupling. Revisit if the verify logic grows past that threshold.

**Why not sign releases with Sigstore at v1?** The spec lists signing as out of scope. Adding attestations is additive and can be introduced without breaking existing installs — a later ADR can turn it on.

**Why Python 3.12 for the build-and-check job?** Any supported version works; picking the middle of the matrix keeps the build reproducible across contributor machines. Python's `build` package and hatchling are version-agnostic.
